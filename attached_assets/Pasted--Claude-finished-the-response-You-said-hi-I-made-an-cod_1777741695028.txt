Two solid approaches for this, depending on how deep you want Neovim integration:

Option A — Spawn Neovim via RPC (nvim-rs)
This is full Neovim embedded in your TUI. You talk to it over msgpack RPC.

toml
# Cargo.toml
[dependencies]
ratatui = "0.29"
crossterm = "0.28"
nvim-rs = { version = "0.6", features = ["use_tokio"] }
tokio = { version = "1", features = ["full"] }
rust
// Spawn neovim with --embed, communicate via stdin/stdout RPC
use nvim_rs::{create::tokio as create, rpc::handler::Dummy};

let handler = Dummy::new();
let (nvim, io_handler) = create::new_child_cmd(
    tokio::process::Command::new("nvim")
        .arg("--embed")
        .arg("--headless"),
    handler,
).await.unwrap();

// Open a file, get buffer content, set lines, etc.
nvim.command("e src/main.rs").await?;
let buf = nvim.get_current_buf().await?;
let lines = buf.get_lines(&nvim, 0, -1, false).await?;
Then in Ratatui you render the buffer lines yourself and sync changes both ways.

Option B — Minimal: Hand off to Neovim (simplest, production-practical)
Most real coding agents (Aider, etc.) do this — leave raw terminal mode, open Neovim, user edits, come back.

rust
use std::process::Command;
use crossterm::{execute, terminal};

fn open_in_neovim(path: &str) -> std::io::Result<()> {
    // Suspend TUI
    terminal::disable_raw_mode()?;
    execute!(std::io::stdout(), terminal::LeaveAlternateScreen)?;

    // Hand off to neovim
    Command::new("nvim").arg(path).status()?;

    // Resume TUI
    terminal::enable_raw_mode()?;
    execute!(std::io::stdout(), terminal::EnterAlternateScreen)?;
    Ok(())
}
Simple, reliable, zero sync complexity.

Minimal Ratatui Layout for a Coding Agent
rust
use ratatui::{
    layout::{Constraint, Direction, Layout},
    widgets::{Block, Borders, Paragraph, List, ListItem},
    style::{Style, Color},
};

fn ui(f: &mut Frame, state: &AppState) {
    let chunks = Layout::default()
        .direction(Direction::Horizontal)
        .constraints([Constraint::Percentage(25), Constraint::Percentage(75)])
        .split(f.area());

    // Left: file tree
    let items: Vec<ListItem> = state.files.iter()
        .map(|f| ListItem::new(f.as_str()))
        .collect();
    f.render_widget(
        List::new(items).block(Block::default().title("Files").borders(Borders::ALL)),
        chunks[0]
    );

    // Right: split agent output + code preview
    let right = Layout::default()
        .direction(Direction::Vertical)
        .constraints([Constraint::Percentage(60), Constraint::Percentage(40)])
        .split(chunks[1]);

    f.render_widget(
        Paragraph::new(state.agent_output.as_str())
            .block(Block::default().title("Agent").borders(Borders::ALL)),
        right[0]
    );

    f.render_widget(
        Paragraph::new(state.code_preview.as_str())
            .block(Block::default()
                .title("Code  [e] edit in nvim")
                .borders(Borders::ALL)
                .border_style(Style::default().fg(Color::Green))),
        right[1]
    );
}
My recommendation
Go with Option B initially — hand off to Neovim cleanly. Option A (nvim-rs RPC) gives you things like real-time buffer sync, inline diffs, and autocomplete integration, but adds significant complexity. You'd only need it if you want the agent to write into Neovim live (like Cursor does) while the user watches.

If that's the goal (agent streams edits directly into the Neovim buffer as the user watches), that's Option A territory — worth it but plan for a few days of nvim RPC debugging.
